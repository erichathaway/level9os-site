1. Executive Summary

Total screens designed: 40 screens (10 per ICP). Recommendation for 8 per ICP is argued in Section 7, but all paths are designed at 10 for Eric's judgment.

The 4 ICP through-lines:

  • Individual: "Can I actually do this alone?" starts with honest scarcity acknowledgment, moves to the $580/month waste anchor, and ends at the free-tier with a working sandbox link. The arc is permission-giving: you can start today, one agent, no credit card.
  • Founder / Small Team: "We're patching it together and I know it" opens with pain recognition, drives hard to the $52,686 proof, runs through the governance chassis and competitor cost demolition ($499 vs $1M), and closes at a contact intake. The arc is validation and urgency: your instinct is right, here's what the delay costs.
  • Growth: "We have tools, but they don't talk to each other" opens with multi-vendor sprawl, shows orchestration architecture, surfaces department wrappers as department-level proof, and closes at the 30/90/180 methodology with a call booking. The arc is consolidation: one control plane replacing five single-vendor patches.
  • Enterprise: "I need to evaluate this without getting locked in" opens with anti-lock-in commitment, surfaces compliance capabilities, runs the full competitive analysis, and closes at a custom evaluation request. The arc is risk removal: we don't ask you to rebuild, we govern what you already have.

Top 5 design decisions proposed:

  1. 1. The diagnostic question ("How big is your operation?") should offer exactly 4 size buckets that map 1:1 to ICPs. No free text. No ambiguity. Auto-routing happens immediately on click. This is the fastest path to personalized content.
  2. 2. Tour buttons should be persistent and visible from the splash state. Not hidden behind a first interaction. They are the direct path and should not require discovery.
  3. 3. The Counter module becomes the ICP anchor for every path at screen 2 or 3. Each ICP gets a different number already coded in ICPCOUNTERFRAMES. This is the most important personalization in the existing codebase.
  4. 4. Group F chips (playful: "Show me the money," "Open the pod bay doors," "Roll the tape," "Houston, we have a problem," "I'll be back") are retired entirely. They were entertainment and they break path coherence.
  5. 5. Voice Pitch module stays in the Founder and Growth paths only, positioned at screen 6-7 as a depth option, not a primary screen. It is transcript-only for now. The "voice render coming soon" banner is acceptable for those ICPs. It does not appear in Individual or Enterprise paths.

3 risks / things to watch:

  1. 1. Path contamination: If a visitor self-selects Founder Tour but then clicks a chip that opens the Architecture module (which is deep / dense), they may lose the emotional arc. The chip discipline ruleset (Section 5) must be enforced in code, not just in design.
  2. 2. Counter module ICP state: The ICPCOUNTERFRAMES are coded but the counter currently needs an ICP value set on the ConversationState. If the direct tour buttons set the ICP but the counter renders before state propagation completes, the visitor sees the default $52,686 frame instead of their personalized number. This is a sequencing bug to watch in implementation.
  3. 3. Breakoff rate at screen 3-4: Most visitors will leave before screen 6 regardless of path. The email/digest capture (positioned at screen 3 in all four paths) is the highest-leverage implementation in this plan. If it ships without that breakoff handling, conversion optimization is flying blind.

4. Probability Ranking Engine (v2 — shipped)

The hard-bucket ICP filter has been replaced with a soft-probability distribution that evolves with every visitor interaction. This section documents the engine as shipped to the rebuild branch (commit 600511a).

State

icpProbability: { solo: number; smb: number; growth: number; enterprise: number } — always sums to 1.0. Persisted in localStorage across sessions. Initial state: uniform {0.25, 0.25, 0.25, 0.25}.

Update rules

Explicit nudge (ICP detection chip click): Named ICP → 0.70, other three ICPs split 0.30 equally. Renormalized to sum 1.0.

Implicit nudge (any chip with icpAffinity): Bayesian update at learning rate 0.05. new_prob[k] = (1 - 0.05) * current[k] + 0.05 * affinity_normalized[k]. Small shift toward chip's affinity profile accumulates over multiple clicks.

Decay (anti-lock-in): Fires every 10 clicks and every 5 minutes of idle. Each tick: 5% drift back toward uniform. Prevents a misclick from permanently biasing the distribution.

Chip scoring

score = affinityDotProduct + recencyPenalty + closedTabBoost + universalBonus

affinityDotProduct = sum(chip.icpAffinity[k] * icpProbability[k])
recencyPenalty     = -0.3 if chip shown in last 10, else 0
closedTabBoost     = +0.2 if chip.opensModule in closedTabs, else 0
universalBonus     = +0.1 if chip.universal === true, else 0

Diversification (slot 4 — "we leave a way out")

Slots 1-3: top 3 chips by score. Slot 4: highest-scoring chip whose dominant ICP is different from slot 1's dominant ICP. A visitor inferred as solo always sees one growth/enterprise chip. Prevents the interface from collapsing entirely into one segment's voice before explicit self-identification.

ICP-first on screen 1 (v3 update)

For fresh sessions with no modules unlocked and no ICP set: ICP detection chips always appear first. The engagementLevel < 2 universal-first override was removed in v3 Phase 1 (commit 248b10e). Universal chips surface from Screen 2 onward via the probability engine ranking with their universalBonus intact. ICP detection chips always have priority on Screen 1.

Dev debug overlay

Set NEXT_PUBLIC_DEBUG_ICP=true in .env.local to enable a fixed bottom-right overlay showing live probability percentages and dominant ICP. Hidden in production.

5. Pre-Built Tour Entry Mechanic

Where the tour buttons live:

The four tour buttons ("Founder Tour," "Growth Tour," "Enterprise Tour," "Individual Tour") should be visible in the splash state before any first interaction. Not after a first message. Not after the diagnostic. From the first second.

Placement: below the MAX input field, in a horizontal row of four pill-shaped buttons. They sit between the input field and the suggested chips row. On mobile, they stack 2x2.

They remain visible throughout the dashboard state as a collapsible row. Collapsed by default after the visitor has been in the path for more than 3 screens. A "Change tour" chip at the bottom of the chip set re-expands them.

What happens when clicked:

Clicking a tour button:


  1. 1. Sets the ICP state immediately (solo / smb / growth / enterprise)

  2. 2. Injects a MAX response: the Screen 1 agent response for that ICP (see Section 3)

  3. 3. Surfaces the Screen 1 module immediately without waiting for a chip click

  4. 4. Advances the chip set to the Screen 1 chip options for that ICP

The visitor does not need to click a chip to see content. The button click is the first chip click. Screen 1 content is visible immediately.

Interaction with the diagnostic path:

The diagnostic path ("How big is your operation?") is the default first-time experience when no tour button has been clicked. It offers 4 size buckets:


  • "Just me or one other person" (routes to Individual)

  • "10 to 50 people" (routes to Founder/SMB)

  • "50 to 200 people" (routes to Growth)

  • "More than 200 people" (routes to Enterprise)

If the visitor has clicked a tour button, the diagnostic is skipped. The tour button is the explicit override.

If the visitor completed the diagnostic and was routed, but then clicks a different tour button, the ICP state resets and the new path starts from Screen 1 with a brief MAX note: "Switching to the [X] tour. Here's where we start."

The diagnostic and the tour buttons coexist. The diagnostic is the passive path for visitors who don't self-identify. The tour buttons are the active path for visitors who know who they are.

Visual treatment of the tour buttons:

Each button uses the ICP accent color at low opacity with a colored left border accent:


  • Individual: emerald #10b981 background at 8% opacity, emerald border

  • Founder: red #ef4444 at 8%, red border

  • Growth: violet #8b5cf6 at 8%, violet border

  • Enterprise: cyan #06b6d4 at 8%, cyan border

Button text is monospace, small caps, 11px. Label format: "Individual Tour" / "Founder Tour" / "Growth Tour" / "Enterprise Tour". No emoji. No icons. Clean label only.

On hover, border brightens to 40% opacity and a subtle MagneticButton pull effect is appropriate here since the brand package exposes it.

8. Open Questions for Eric (v3 updated)

1. 10 screens per ICP or 8?

Recommendation: 8. Here is the rationale.

Screen 8 in every path is where conversion pressure arrives ("How do we start?"). Screens 9 and 10 are post-decision cleanup for visitors who need confidence, not conversion. Analytics will show that most visitors who reach screen 8 either convert or leave within the next 2 interactions. Screens 9 and 10 are serving a minority.

The 10-screen design is presented in full here because you asked for it and it ensures nothing important is missing from the arc. But in implementation, screens 9 and 10 of each ICP should be positioned as optional depth screens behind a "Tell me more first" chip, not as mandatory steps in the path progression.

The counter-argument for 10: About and Try It (screens 9-10 for Individual) are important conversion elements and should not be buried. For Individual specifically, keeping the full 10 is justified because the conversion moment (free-tier signup) benefits from the trust-building at screen 8 (founder story).

2. Should ICPs share screens?

Recommended: Yes for Why Us Race and Compounding Risk. No for Counter, Live Feed, and Calculator.

Why Us Race appears in Founder (screen 6), Growth (screen 6), and Enterprise (screen 6). That's fine. The module is generic enough. For Growth, the competitor set should swap Workday for "Zapier/n8n/Make" but this is a prop change, not a new module.

Compounding Risk appears in Founder (screen 3), Growth (screen 1), and as a supporting chip in Enterprise. That's also fine. It's the universal visual proof of the core problem.

Counter, Live Feed, and Calculator should NOT be shared across paths without ICP adaptation. Counter already has ICP frames coded. Live Feed needs the Enterprise compliance events added. Calculator needs ICP-aware default seeding. If these appear in multiple paths, they must render with ICP context.

3. Diagnostic AND direct tour buttons, or one or the other?

Recommended: Both, with the direct tour buttons visible from the first second.

The diagnostic path is valuable for visitors who arrive without context and don't know which ICP they are. The direct tour buttons are valuable for visitors who self-identify immediately. These are complementary, not competitive.

The visual risk is clutter. The splash state currently has: MAX message, input field, suggested chips. Adding 4 tour buttons below the input could crowd the first impression. A clean solution: the 4 tour buttons live in a separate row labeled "Pick your path:" at 11px monospace above the chips. They are visually distinct from the chips (pill border style vs chip background style).

If Eric finds the diagnostic redundant because the tour buttons cover the same routing, the diagnostic can be demoted to screen 1 of the "no tour selected" path only, which means visitors who arrive cold see one diagnostic question, and visitors who click a tour button skip it entirely.

4. Should Voice Pitch surface in any guided path now?

Recommended: Yes, in Founder and Growth only, as a depth option at screens 6-7. Position it as "Hear the 1:30 pitch" not "Roll the tape."

The transcript-only state is acceptable for these ICPs because they are more patient than Individual and more open to depth than Enterprise. The "Voice render coming soon" banner is honest and does not damage credibility if the surrounding content is strong.

Voice Pitch should NOT surface in Individual (too long, wrong format for the attention span) or Enterprise (transcript view does not meet the visual quality bar for an enterprise evaluation context).

7. Does the Architecture module belong in Founder path?

The current design puts Architecture in the Growth and Enterprise paths but cuts it from Individual and Founder. The counter-argument: the "plug any agent" concept is the Founder ICP's most important "why not just use Claude" answer, and Architecture is where that answer lives most completely.

Decision point: if the Governance module carries the "plug any agent, one control plane" message effectively (it does, via the multi-vendor orchestration capability card), then Architecture is not needed in the Founder path and should stay cut. If the Founder ICP frequently asks "but how does this actually work under the hood," Architecture at screen 7 or 8 as an optional depth chip would address that without making it a mandatory screen.

8. Should diversification be suppressed after explicit ICP self-identification? (new — v2)

The diversification slot (slot 4) will sometimes show a chip aimed at a segment the visitor self-identified away from (e.g., an enterprise chip for a visitor who clicked "Just me"). This is intentional per spec ("we leave a way out") but may feel inconsistent to visitors who self-identified clearly. Suppressing it after an explicit nudge is a 2-line change. Recommendation: leave active and observe in real sessions first.

9. Learning rate 0.05 — is that too slow for short sessions? (new — v2)

At 0.05 learning rate, it takes approximately 8-10 chip clicks for a strong affinity signal to meaningfully shift the distribution. If average session depth is fewer than 5 clicks, consider raising to 0.10 or 0.15. Monitor via debug overlay + session analytics before locking the rate.

10. Should the probability distribution be reflected in any visible UI element? (new — v2; deferred to v4)

Not in production. But a subtle accent color shift as probability builds could reinforce that the interface is adapting to the visitor. Out of scope for v3. Flag for v4 consideration.
